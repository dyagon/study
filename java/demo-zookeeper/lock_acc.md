

# 代码问题

这段代码演示了如何使用 ZooKeeper 分布式锁来保证线程安全。虽然它在逻辑上可以跑通（即能在单机上得到正确结果），但在**架构设计**和**演示有效性**上存在几个关键问题，容易给初学者带来误导。

以下是详细的问题分析：

### 1. 架构反模式：用“分布式锁”解决“单机问题” (最大的问题)

代码中使用 ZooKeeper 锁（`ZkLockV2`）去保护一个 **JVM 内部的内存变量** (`SharedResource.value`)。

*   **为什么这是错的？**
    *   **分布式锁的用途：** 是为了协调**跨进程（跨机器）** 对 **外部共享资源**（如数据库的一行记录、Redis 的一个 Key、一个文件）的访问。
    *   **本地锁的用途：** JVM 内部的内存变量（如本例中的 `value`），应该使用 Java 自带的 `synchronized` 关键字、`ReentrantLock` 或 `AtomicInteger` 来保护。
*   **后果：**
    *   **性能极差：** 访问本地内存只需要纳秒级时间，而 ZK 锁涉及网络通信、磁盘写入（ZK 事务日志），耗时是毫秒级的。为了保护一个简单的 `int++` 操作，引入了巨大的网络 I/O 开销。
    *   **无法扩展：** 如果你启动两个这个 Java 程序的实例（模拟两个微服务节点），虽然它们会争抢同一个 ZK 锁，但它们各自维护了一个 `SharedResource` 实例。节点 A 锁住了，改的是节点 A 的内存；节点 B 锁住了，改的是节点 B 的内存。**数据依然是不一致的**。
    *   **正确演示做法：** 分布式锁的 Demo 应该去操作一个**公共存储**（例如，把计数器存在 Redis 里，或者存在 ZK 的节点数据里），这样才能体现出“分布式”的意义。

### 2. 性能陷阱：高频短任务

*   **场景：** 这里的业务逻辑极其简单（累加 1），循环次数也较多（每线程 100 次）。
*   **问题：** 这是一个典型的**细粒度锁（Fine-grained Locking）** 场景。
    *   线程每执行一次 `resource.increment()` 就要进行一次完整的 `Lock -> Network -> Unlock` 流程。
    *   这意味着 1000 次累加操作会触发 **2000 次 ZK 网络请求** 和 **1000 次 ZK 临时节点的创建与删除**。这不仅运行速度极慢，而且对 ZK 集群造成了不必要的压力。
*   **优化建议：** 在分布式场景下，通常建议适当增大锁的粒度（例如获取一次锁，处理批量任务，再释放），或者使用 Redis 等更高吞吐的中间件。

### 3. 对象生命周期管理

*   **代码现状：**
    ```java
    // 在循环外部创建锁对象？不，是在任务内部
    executor.submit(() -> {
        ZkLockV2 lock = new ZkLockV2(client, LOCK_PATH); // 每个线程创建一个锁对象
        // ...
    });
    ```
*   **分析：**
    *   **优点：** 这种写法**巧妙地规避了**上一段代码（`ZkLock.java`）中存在的“线程安全 Bug”。因为每个线程都有自己独立的 `ZkLockV2` 实例，它们不共享 `currentNodePath` 等状态，只在 ZK 服务端竞争。所以这段代码**能跑通**，不会报错。
    *   **缺点：** 如果 `ZkLockV2` 实现中有一些比较重的资源初始化（比如内部又创建了线程池或 heavy Watcher），频繁 `new` 对象会有开销。但在 Curator 封装下，主要开销还是在网络。

### 4. 异常处理的隐患

在 `lock.lock()` 和 `lock.unlock()` 的使用上，虽然用了 `try-finally`，但有一个细节需要注意：

```java
lock.lock(); // 如果这里抛出异常（比如网络断了）
try {
    resource.increment();
} finally {
    lock.unlock(); // 这里会执行
}
```

如果 `lock.lock()` 本身因为网络原因抛出异常（没有加锁成功），程序会直接跳到 `finally` 块执行 `unlock()`。
*   如果 `unlock()` 内部没有判断“当前是否持有锁”（即 `locked` 状态），直接去调用 ZK 删除节点，可能会抛出 `IllegalMonitorStateException` 或掩盖原本的异常。
*   **建议：** 标准写法通常是将 `lock.lock()` 放在 `try` 块**之前**。

```java
lock.lock(); // 加锁成功后再进 try
try {
    resource.increment();
} finally {
    lock.unlock();
}
```

### 总结

这段代码是一个**合格的功能演示**，但是一个**错误的架构示范**。

*   **它的优点：** 正确使用了 Curator 客户端，展示了加锁和不加锁的结果差异。
*   **它的问题：** 混淆了“本地并发安全”和“分布式并发安全”的界限。如果面试中拿出这段代码说“这是我的分布式锁应用场景”，会被认为不懂分布式系统的基本原理。


# 代码问题 2

你的观察非常敏锐！结果是 **100** 而不是其他随机数字（比如 532 或 890），这确实是一个非常极端的竞态条件（Race Condition）现象。

这并不是说“只有一个线程在工作”，而是**所有 10 个线程都在工作，但是它们在“齐步走”，互相覆盖了彼此的工作**。

造成这种“每次都是 100”的罪魁祸首是那行 `Thread.sleep(1)`。让我们慢动作回放一下发生了什么。

### 核心原因：人工制造的“完美并发”

`SharedResource.increment()` 的代码逻辑是这样的：

```java
// 1. 读取（Read）
int temp = value; 
// 2. 等待（Sleep）- 这里的 1ms 对 CPU 来说是沧海桑田
try { Thread.sleep(1); } catch... 
// 3. 写入（Write）
value = temp + 1; 
```

当 10 个线程并发启动时，因为有了 `sleep`，发生了可怕的**步调一致**：

#### 第一轮循环 (i=0)
1.  **读取阶段：**
    *   线程 T1 启动，读取 `value`，此时是 **0**。记在自己的小本本上 `temp = 0`。
    *   线程 T1 进入睡眠。
    *   **关键点：** 因为 T1 睡着了，没有立刻更新 `value`。此时 `value` 还是 0。
    *   线程 T2 启动，读取 `value`，依然是 **0**！`temp = 0`。T2 也睡了。
    *   ...
    *   线程 T10 启动，读取 `value`，还是 **0**！`temp = 0`。T10 也睡了。
    *   **结果：** 10 个线程手里拿到的旧值（temp）全都是 0。

2.  **写入阶段：**
    *   1 毫秒后，大家陆续醒来。
    *   T1 计算 `0 + 1 = 1`，把 `value` 改为 **1**。
    *   T2 计算 `0 + 1 = 1`，把 `value` 改为 **1**（覆盖了 T1 的工作）。
    *   ...
    *   T10 计算 `0 + 1 = 1`，把 `value` 改为 **1**（覆盖了前面的工作）。

**第一轮结束后，虽然 10 个线程都累加了一次，但 `value` 仅仅变成了 1（而不是期望的 10）。**

#### 第二轮循环 (i=1)
*   线程 T1 读取 `value`，拿到 **1**。进入睡眠。
*   线程 T2 读取 `value`，拿到 **1**。进入睡眠。
*   ...
*   大家醒来，都把 `value` 更新为 **2**。

#### 第 100 轮循环
*   大家一起读取 **99**。
*   大家一起写入 **100**。

---

### 为什么结果这么“整齐”？

如果没有 `Thread.sleep(1)`，CPU 执行速度极快，线程间的切换是随机的，结果通常会是一个 100 到 1000 之间的随机数（例如 874）。

但是加上了 `sleep(1)`，相当于给所有线程加了一个**同步器**。这 1 毫秒的空闲时间足够让其他所有 9 个线程都完成“读取”操作。这就导致了**100% 的读写冲突率**。

### 形象的比喻

想象一下大家在同一个黑板上做数学题：

1.  黑板上写着 **0**。
2.  如果不睡觉（正常情况）：
    *   A 看了眼 0，擦掉写上 1。
    *   B 看了眼 1，擦掉写上 2。
    *   C 手慢了，刚看还是 0，擦掉写上 1（这就把 B 的 2 给覆盖回去了）。
    *   结果是乱序的。

3.  如果睡觉（你的代码）：
    *   A 看了眼 0，**去睡了一觉**。
    *   B 趁 A 睡觉，看了眼黑板，豁，还是 0，**也去睡了一觉**。
    *   10 个人都看了一眼 0，都去睡觉了。
    *   大家醒来，每个人都觉得自己应该写 1。
    *   于是 10 个人排队上去把黑板擦了，写上 1。
    *   最终黑板上就是 1。

这就是为什么你得到的总是 **100** —— **所有的并发工作都被完全抵消了，效果等同于只有一个线程在运行。** 这正是演示“为什么我们需要锁”的绝佳反面教材。
